VERDICT: Transplant-ready with moderate confidence; critical prompt content issues and an incomplete experimental pipeline need fixing before non-science lanes will produce reliable, stylistically contained episodes.

NEW FINDINGS (things the local panel plausibly missed):
- [content quality] The “forbidden patterns” metadata approach is impotent for guiding LLM generation. In `preview.py` the forbidden plot patterns are rendered only as a count—never as concrete prohibitions the model can obey (line: `"Forbidden pattern count: {len(profile.forbidden_plot_patterns)}"`). The TEMPLATE_SEAMS allowlist (`contracts.py`) does not include `forbidden_plot_patterns` or `tone_guardrails`, so even if a pack author wanted to inject them into a prompt, the registry would reject the variable. As a result, packs like `media_restoration_adventure` tell the LLM nothing about avoiding “Star‑Trek‑style mission plot” or “laboratory containment breach”; the model may freely generate those patterns, and the leakage scan only detects them post‑hoc. Fix: amend `LABEL_TEMPLATE_VARIABLES` to include `forbidden_plot_patterns` and `tone_guardrails`, and require packs to weave them into a suitable system prompt (e.g. `outline_system`).  
- [design‑level / runtime] The `simple_4_prompt_experimental` pipeline will fail with a real local LLM because it lacks any mechanism to convey the source material, tone guardrails, or ledger schema to the model. The runner (`runner.py`) passes only the bare prompt string from the pack; it does not inject the StoryInputPacket’s text, the profile’s constraints, or any schema documentation. The pack’s prompts, e.g. `pass_1_creative_story`, are abstract and contain no inline guardrails, so the LLM operates in a vacuum. In production this pipeline is invisible, but the lab advertises it as an executable experiment—users (including Jeffrey) may expect it to work. Fix: either (a) enrich `run_pipeline` to accept a full context dict and expand template-like placeholders, or (b) retire the executable flag and clearly mark the 4‑pass pipeline as “schema placeholder only, not functional”.  
- [spec‑level contradiction / missing validation] The `legacy_many_pass` pipeline declares passes (`style_pick_inventor`, `style_pick_chooser`) that reference seams which the `science_news_default` pack does not supply. While the bank’s `required_seams` omit those, the pipeline’s `passes` still declare them. The production modules handle this by falling back to empty strings (retaining byte‑identical defaults), but the lab’s own validator and bridge make no cross‑check that a pack’s actual `prompt_stages` cover every seam a pipeline’s passes require. If someone later adds a new executable pipeline that references a seam missing from an existing pack, the failure would only surface at runtime inside the runner (or worse, silently in production if the pack happens to have an empty string that is handled gracefully). This is a subtle coupling violation: pipelines and packs are validated atomically, not in combination. Fix: add a registry cross‑validation that, for each `(bank, model, pipeline)` combo, the pack contains every seam referenced by the pipeline’s `seam_refs` (or at least warn for non‑executable pipelines, hard‑error for executable ones).

CONTENT CRITIQUE (the story‑pack prose itself):
- `media_restoration_adventure` / `gentle_thriller`: The outline systems are far too generic (“Build the episode from damaged media…”, “Build suspense from time pressure…”). They never show the LLM a worked example of a complete outline, nor do they concretize what “preservation labour” or “fragile evidence” translates to in terms of beat structure. As a result, an offline 7B–13B model is likely to produce samey, scene‑less summaries. I would rewrite the outline prompt to include a short hand‑crafted fragment: for example, a three‑beat outline skeleton with character names, a physical object that drives the action, and a clear before‑after transformation in the archive world.
- `faithful_radio_adaptation`: The instruction “Compression is allowed; replacement is not.” is the right spirit, but the prompt lacks any heuristic for how to compress. Without a compression strategy (e.g., “combine two chapters into a single scene by merging characters’ dialogues while keeping the core decision point intact”), the model may either truncate haphazardly or, worse, fabricate connective tissue that replaces the source. A concrete compression recipe as a system instruction would dramatically raise output quality.
- `archival_documentary` motion prompts: The prose reads like set‑dressing descriptions rather than motion‑first prompts. For `ltx‑video` a cue like “Continuous shot, same catalog table throughout. Slow documentary push‑in across labeled archive boxes. Paper labels flutter faintly. Dust motes drift in window light.” risks the model fixating on the nouns and delivering a still shot. Production’s `_LTX_MOTION_PROMPT_BY_ROLE` table is famously noun‑sparse, using pure motion verbs and camera directions. I would rewrite the motion prompts to foreground camera movement, e.g.: *announcer* → “Steadicam slow push‑in, shallow rack focus, gentle lateral drift.” Keep the long‑shot continuity (“same table throughout”) as a directive but strip the object flourishes.
- `simple_4_prompt_experimental`: The four passes are completely detached from the actual ledger schema. “Fill the production story ledger with cast, lines, beats…” without any schema documentation means that only a very large, instruction‑tuned model that already knows the OTR ledger format could possibly succeed. For this to be a real experiment, the pack must include a condensed version of the required JSON schema, or the runner must inject it. Until then it is a fiction.

ABSENT THINGS:
- **Re‑pinning guide for `compat.py`.** When production refactors and the AST drift tests fail, Jeffrey needs a step‑by‑step checklist: how to refresh the mirror, how to run the extraction scripts, exactly which constants to update, and a verification command. Without this, the “loud failure” becomes a panic with no obvious recovery path.
- **Operator‑scale smoke test for real LLM output.** The validator and FakeLLM tests prove structural validity, but nothing exercises the system end‑to‑end with the actual offline LLMs Jeffrey will use (e.g., Llama 3, Mistral). A minimally‑scripted run using a real engine for one pack, with a quality‑checklist pass/fail, would surface dramatic weakness early.
- **Forbidden‑pattern prompt integration.** As noted above, the architecture lacks a channel for delivering explicit negative constraints into the LLM. Adding a `negative_constraints_section` seam or simply making the variables available is the smallest version worth adding.
- **Operator documentation for adding a new bank/pack/style.** The `CUSTOM_SOURCE_BANK_GUIDE.md` is referenced but not attached; given the rigorous validation, a novice‑friendly guide is essential to prevent 1am copy‑paste accidents from introducing contradictory forbidden terms or missing seams.

DO-NOT-DO:
- **Remove `simple_4_prompt_experimental` from the transplant until it is made functional.** Its presence as an `executable_in_lab: true` pipeline with no prompt enrichment will only mislead operators into thinking they can generate a valid episode in four steps. Either sink it to a documentation‑only stub or complete the injection mechanism; shipping it half‑built is worse than delaying it.
